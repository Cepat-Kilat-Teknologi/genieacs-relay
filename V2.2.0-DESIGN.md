# genieacs-relay v2.2.0 — Design Document

> **Status:** Draft, awaiting implementation across 4 phases.
> **Driver:** auto-learning OLT deployments (Hioso, HSGQ, Jolink,
> CDATA auto-mode, etc.) push **all** customer-facing config to the
> CPE side via TR-069, so the agent needs a much richer set of CPE
> management endpoints than the v2.1.0 CRUD slice currently exposes.
> See `~/Projects/isp-agent` discussion thread session 5h-followup
> for the gap analysis that motivated this document.

## Goals

1. **Cover the operational surface** an ISP NOC needs for daily
   CPE management when the OLT is in auto-learning mode and cannot
   itself push profile config. The agent must do everything via
   TR-069 from the GenieACS plane.
2. **Ship in a backwards-compatible MINOR release** (v2.1.0 →
   v2.2.0). Every existing endpoint stays exactly as-is. New
   endpoints add to the surface, do not modify it.
3. **Maintain 100% main-package coverage** per the existing CLAUDE.md
   discipline. Every new handler, domain function, and helper has
   matching tests.
4. **Stay idiomatic** with the existing `package main` flat layout,
   the chi router middleware chain, the `sendResponse / sendError`
   envelope pattern, and the `ExtractDeviceIDByIP` helper convention.

## Non-goals

1. **Not** a TR-069 ACS rewrite. We delegate to GenieACS NBI for
   everything; this is just a thin REST wrapper that hides the NBI
   query language and TR-069 task semantics from the agent.
2. **Not** vendor-specific provisioning DSLs. Where TR-098 vs TR-181
   path differences exist (e.g. PPPoE under
   `WANDevice.*.WANConnectionDevice.*.WANPPPConnection.*` vs
   `Device.PPP.Interface.*`) we pick a sensible default and document
   the alternative as a v2.3+ vendor-detection enhancement, mirroring
   how `getOpticalStats` handles 5-vendor optical parameter paths
   today.
3. **Not** a stateful workflow engine. Long-running operations
   (firmware download) return the GenieACS task ID and let the agent
   workflow side handle retry / polling / saga compensation. We do
   NOT introduce any persistent task store in the relay.

## Versioning impact

- v2.1.0 (current) — 14 operational endpoints under `/api/v1/genieacs`
- **v2.2.0 (this design)** — 14 + **25 new** = 39 operational endpoints
- `isp-agent v0.2.0` will bump `depends_on: genieacs-relay >=2.2.0`
- All v2.1.0 endpoints remain unchanged; this is strictly additive

## Conventions used in this document

- **Method/Path** — relative to `/api/v1/genieacs`
- **Idempotency tier** — Read (no idempotency middleware needed; safe to
  retry); Write (X-Idempotency-Key middleware applies); LongRunning
  (returns 202 + task_id, polling separate)
- **Vendor floor** — minimum CPE TR-069 dialect supported; "TR-098" =
  IGD model (older, more common in EPON/GPON ZTE/Huawei consumer
  ONUs), "TR-181" = Device:2 model (newer, broadband forum standard)

## Phase 1 — Structural foundations

### 1.1 `tr069.go` — generic TR-069 RPC helpers

A new file `tr069.go` exposes the TR-069 RPC verbs that GenieACS NBI
supports as a `name` field on the `/devices/{id}/tasks` endpoint, but
that the relay does not yet wrap. Each helper takes a `context.Context`
and a `deviceID` and submits the appropriate JSON task body via
`postJSONRequest`. Errors are wrapped with the task name for
diagnosability. All helpers follow the existing `rebootDevice`
pattern at `reboot.go:22`.

Helpers added:

```go
// FactoryReset — TR-069 FactoryReset RPC. Destructive.
func factoryResetDevice(ctx context.Context, deviceID string) error

// DownloadRequest is the TR-069 Download RPC parameter set.
type DownloadRequest struct {
    FileType        string  // e.g. "1 Firmware Upgrade Image"
    URL             string
    Username        string  // optional HTTP basic auth
    Password        string  // optional HTTP basic auth
    FileSize        int     // optional, in bytes
    TargetFilename  string  // optional
    DelaySeconds    int     // optional, schedule delay
    SuccessURL      string  // optional, callback on success
    FailureURL      string  // optional, callback on failure
    CommandKey      string  // optional, NBI client correlation
}

// downloadFile — TR-069 Download RPC (firmware/config push). Long-running.
func downloadFile(ctx context.Context, deviceID string, req DownloadRequest) (taskID string, err error)

// getParameterValues — TR-069 GetParameterValues RPC. Returns the
// task ID; the actual parameter values arrive on the device tree
// after the device next informs the ACS.
func getParameterValues(ctx context.Context, deviceID string, paths []string) (taskID string, err error)

// addObject — TR-069 AddObject RPC. Creates a new instance under
// the given object path. Returns the new instance number on success.
func addObject(ctx context.Context, deviceID string, objectName string) (instance int, err error)

// deleteObject — TR-069 DeleteObject RPC. Removes the instance.
func deleteObject(ctx context.Context, deviceID string, objectName string) error

// connectionRequest — fires a connection request without queuing a
// task. Used by the H2 wake endpoint. Wraps GenieACS's
// `POST /devices/{id}/tasks?connection_request` with an empty task
// list (or a no-op getParameterValues for [_lastInform]).
func connectionRequest(ctx context.Context, deviceID string) error
```

Each helper has matching unit tests in `tr069_test.go` covering:
happy path (200), happy path (202 — async queue), 4xx error, 5xx error,
network failure, invalid GenieACS URL, body decode error.

### 1.2 `param_walker.go` — typed parameter tree walker

The existing `getDeviceData` returns `map[string]interface{}` which
is awkward to walk with type safety. New helpers:

```go
// LookupString walks a dotted parameter path and extracts the _value
// field as a string. Returns ("", false) if missing or wrong type.
func LookupString(tree map[string]interface{}, path string) (string, bool)

// LookupInt — same, for int values
func LookupInt(tree map[string]interface{}, path string) (int, bool)

// LookupBool — same, for bool values
func LookupBool(tree map[string]interface{}, path string) (bool, bool)

// LookupTime — same, for ISO-8601 timestamps
func LookupTime(tree map[string]interface{}, path string) (time.Time, bool)

// EnumerateInstances finds all numeric instance keys under a parent
// path, e.g. WANDevice.{1,2,3}.WANConnectionDevice — returns sorted
// instance numbers.
func EnumerateInstances(tree map[string]interface{}, parentPath string) []int
```

Used by H1 (`/status`), H4 (`/wan`), H7 (`/params`), and most of the
MEDIUM and LOW endpoints. Centralizes the "walk a Genie tree" logic
so we don't open-code it 25 times.

### 1.3 IP-to-device-id direct passthrough mode

The existing `ExtractDeviceIDByIP` always resolves IP to device ID
via the `/devices/?query=...` NBI lookup. That's correct for
99% of cases — but the deferred `ClearDeviceCache` workflow needs
the relay to accept a `device_id` directly because the underlying
GenieACS endpoint uses `device_id` as its addressing key.

We add a sibling helper `ExtractDeviceIDOrPassthrough` that:

1. If the URL parameter is named `device_id`, use it directly
2. If the URL parameter is named `ip`, run the existing
   `getDeviceIDByIP` resolver

This unblocks `ClearDeviceCache` and any future endpoint that wants
to address by either key. New endpoints in v2.2.0 default to `{ip}`
for consistency with the existing 14.

## Phase 2 — HIGH priority endpoints (7)

### H1 · GET /status/{ip}

| Field | Value |
|---|---|
| Use case | "Is the customer online right now?" — operational dashboard, customer support first-line. Returns last inform timestamp, computed online flag, uptime in seconds. |
| Idempotency tier | Read |
| TR-069 RPC used | None — derives from cached device tree |
| Vendor floor | TR-098 + TR-181 (parses both) |

**Request:**
```
GET /api/v1/genieacs/status/192.168.1.1
```

**Response 200:**
```json
{
  "code": 200,
  "status": "success",
  "data": {
    "device_id": "001141-F670L-ZTEGCFLN794B3A1",
    "ip": "192.168.1.1",
    "last_inform": "2026-04-14T11:35:21Z",
    "last_inform_age_seconds": 42,
    "online": true,
    "uptime_seconds": 1234567,
    "uptime_human": "14 days, 6 hours",
    "model": "F670L",
    "manufacturer": "ZTE",
    "software_version": "V9.0.10P5N12",
    "hardware_version": "V1.0"
  }
}
```

**Online classification:** `online = last_inform_age_seconds < 3 × stale_threshold`. Configurable via the existing `STALE_DEVICE_THRESHOLD_SECONDS` env var that already gates `getDeviceIDByIP`.

**Errors:** 404 NOT_FOUND (device unknown / stale).

### H2 · POST /wake/{ip}

| Field | Value |
|---|---|
| Use case | Force a CPE to come online for first-config push (just installed) or to wake an idle device for diagnostics. Fire-and-forget — does NOT block waiting for the device to actually wake up. |
| Idempotency tier | Write |
| TR-069 RPC used | ConnectionRequest |
| Vendor floor | TR-098 + TR-181 |

**Request:**
```
POST /api/v1/genieacs/wake/192.168.1.1
```
(empty body)

**Response 202:**
```json
{
  "code": 202,
  "status": "success",
  "data": {
    "message": "ConnectionRequest dispatched to device. Wake-up takes 1-30 seconds depending on CPE responsiveness."
  }
}
```

**Errors:** 404 NOT_FOUND, 503 SERVICE_UNAVAILABLE (NBI unreachable).

### H3 · PUT /pppoe/{ip}

| Field | Value |
|---|---|
| Use case | Set/update PPPoE credentials on the CPE side. Critical for ISP deployments where PPPoE termination happens on CPE rather than on the BNG (some Indonesian ISPs use this model so the BNG can be a generic L3 router instead of a PPPoE-aware BRAS). |
| Idempotency tier | Write |
| TR-069 RPC used | SetParameterValues |
| Vendor floor | TR-098 (TR-181 path documented but not auto-detected in v2.2.0; vendor detection in v2.3.0) |

**Request:**
```
PUT /api/v1/genieacs/pppoe/192.168.1.1
Content-Type: application/json

{
  "username": "pppoe-customer-001",
  "password": "secret-pass",
  "wan_instance": 1
}
```

`wan_instance` is optional and defaults to **1** (the first WANPPPConnection on the first WANConnectionDevice on the first WANDevice). Caller can override if device exposes multiple PPPoE connections.

**Response 202:**
```json
{
  "code": 202,
  "status": "success",
  "data": {
    "message": "PPPoE credentials updated. Device will reconnect within 30s.",
    "device_id": "001141-F670L-...",
    "ip": "192.168.1.1",
    "wan_instance": 1
  }
}
```

**Validation:**
- `username` non-empty, max 64 chars, no whitespace
- `password` non-empty, min 1 char, max 64 chars
- `wan_instance` 1-8 if specified

**Errors:** 400 VALIDATION_ERROR, 404 NOT_FOUND, 500 GENIEACS_ERROR.

**Path written:**
```
InternetGatewayDevice.WANDevice.{wan_instance}.WANConnectionDevice.1.WANPPPConnection.1.Username
InternetGatewayDevice.WANDevice.{wan_instance}.WANConnectionDevice.1.WANPPPConnection.1.Password
```

### H4 · GET /wan/{ip}

| Field | Value |
|---|---|
| Use case | "Is the WAN link up? What public IP does the customer have?" — first-line debugging, customer self-service "my internet is down". |
| Idempotency tier | Read |
| TR-069 RPC used | None — derives from cached device tree |
| Vendor floor | TR-098 (walks both PPP and IP modes) |

**Request:**
```
GET /api/v1/genieacs/wan/192.168.1.1
```

**Response 200:**
```json
{
  "code": 200,
  "status": "success",
  "data": {
    "device_id": "001141-F670L-...",
    "ip": "192.168.1.1",
    "wan_connections": [
      {
        "instance": 1,
        "type": "pppoe",
        "connection_status": "Connected",
        "external_ip": "203.0.113.45",
        "uptime_seconds": 12345,
        "username": "pppoe-customer-001",
        "last_connection_error": ""
      }

}
```

`wan_connections` is an array because some CPE models expose multiple WANPPPConnection / WANIPConnection instances (multi-WAN, dual-stack v4/v6).

**Errors:** 404 NOT_FOUND.

### H5 · POST /firmware/{ip}

| Field | Value |
|---|---|
| Use case | Push firmware patch to one CPE. Fleet rollout is done by the agent dispatching this endpoint per-device. |
| Idempotency tier | LongRunning |
| TR-069 RPC used | Download |
| Vendor floor | TR-098 + TR-181 |

**Request:**
```
POST /api/v1/genieacs/firmware/192.168.1.1
Content-Type: application/json

{
  "file_url": "https://firmware.example.com/zte-f670l-v9.0.11.bin",
  "file_type": "1 Firmware Upgrade Image",
  "file_size": 12345678,
  "target_filename": "zte-f670l-v9.0.11.bin",
  "username": "",
  "password": "",
  "command_key": "fleet-rollout-2026-04-14-batch-1"
}
```

`file_type` defaults to `"1 Firmware Upgrade Image"` (the most common). Other values: `"3 Vendor Configuration File"`, `"2 Web Content"`, etc. `command_key` is optional metadata.

**Response 202:**
```json
{
  "code": 202,
  "status": "success",
  "data": {
    "task_id": "67abc1234567890abcdef123",
    "message": "Firmware download dispatched. Use GET /api/v1/genieacs/tasks/{task_id} to poll status. Typical duration 60-300 seconds depending on file size and CPE link speed.",
    "estimated_duration_seconds": 180
  }
}
```

`task_id` is the GenieACS NBI task ID, returned verbatim. The companion `GET /tasks/{task_id}` endpoint (added in this design as part of H5) proxies the GenieACS task status.

**Validation:**
- `file_url` must be HTTPS (no plaintext HTTP firmware downloads)
- `file_url` host must not be a private IP / metadata service (basic SSRF guard)
- `file_size` ≥ 0
- `command_key` ≤ 256 chars

**Errors:** 400 VALIDATION_ERROR, 404 NOT_FOUND, 500 GENIEACS_ERROR.

### H6 · POST /factory-reset/{ip}

| Field | Value |
|---|---|
| Use case | RMA flow — wipe a CPE before returning it to inventory. Customer support "reset my modem to factory" requests. **Destructive.** |
| Idempotency tier | Write |
| TR-069 RPC used | FactoryReset |
| Vendor floor | TR-098 + TR-181 |

**Request:**
```
POST /api/v1/genieacs/factory-reset/192.168.1.1
```
(empty body)

**Response 202:**
```json
{
  "code": 202,
  "status": "success",
  "data": {
    "message": "FactoryReset dispatched. Device will be unreachable for 60-180 seconds, will lose its current PPPoE credentials and WLAN config, and will rejoin the ACS with a fresh provisioning state.",
    "device_id": "001141-F670L-...",
    "ip": "192.168.1.1"
  }
}
```

**Errors:** 404 NOT_FOUND, 500 GENIEACS_ERROR.

The relay clears its in-process cache for the device immediately on success since the post-reset device tree will look completely different.

### H7 · POST /params/{ip}

| Field | Value |
|---|---|
| Use case | Generic GetParameterValues passthrough for NOC L2/L3 debugging. Lets operators inspect any TR-069 parameter without the relay needing a dedicated endpoint per parameter. Reduces churn — instead of v2.3.0/v2.4.0 adding 5 more dedicated read endpoints, we ship one generic one. |
| Idempotency tier | Write (uses POST so the request body can carry the path list) |
| TR-069 RPC used | None for cached mode; GetParameterValues for live mode |
| Vendor floor | Any — paths are passed through verbatim |

**Request:**
```
POST /api/v1/genieacs/params/192.168.1.1
Content-Type: application/json

{
  "paths": [
    "InternetGatewayDevice.DeviceInfo.UpTime",
    "InternetGatewayDevice.WANDevice.1.WANConnectionDevice.1.WANPPPConnection.1.Username",
    "InternetGatewayDevice.LANDevice.1.WLANConfiguration.1.SSID"
  ],
  "live": false
}
```

`live=false` (default): walk the cached device tree and return values immediately. Sub-100ms.

`live=true`: trigger a TR-069 GetParameterValues task with `?connection_request`, wait up to 30s for the device to inform back, then read the refreshed tree. Slower but guaranteed fresh.

**Response 200:**
```json
{
  "code": 200,
  "status": "success",
  "data": {
    "device_id": "001141-F670L-...",
    "ip": "192.168.1.1",
    "params": {
      "InternetGatewayDevice.DeviceInfo.UpTime": "1234567",
      "InternetGatewayDevice.WANDevice.1.WANConnectionDevice.1.WANPPPConnection.1.Username": "pppoe-customer-001",
      "InternetGatewayDevice.LANDevice.1.WLANConfiguration.1.SSID": "MyWiFi-2.4GHz"
    },
    "missing_paths": [],
    "live": false
  }
}
```

`missing_paths` lists requested paths that did not exist in the device tree (typo, vendor difference, etc).

**Validation:**
- `paths` non-empty, max 50 entries per request
- Each path matches `^[a-zA-Z][a-zA-Z0-9_.]*$` (no shell metacharacters or query injection)
- `live` is bool, default false

**Errors:** 400 VALIDATION_ERROR, 404 NOT_FOUND, 504 TIMEOUT (live mode).

## Phase 3 — MEDIUM priority endpoints (8)

### M1 · POST /diag/ping/{ip}

Submit TR-069 IPPingDiagnostics task. Returns 202 + task_id; companion poll endpoint reads result. Request body: `{host, count, packet_size, timeout_ms}`.

### M2 · POST /diag/traceroute/{ip}

Same pattern as M1, for TraceRouteDiagnostics. Body: `{host, max_hops, timeout_ms}`.

### M3 · GET /wifi-clients/{ip}

Reads `LANDevice.1.WLANConfiguration.*.AssociatedDevice.*` (TR-098). Returns associated wifi clients per radio with MAC, signal strength dBm, link rate, association time. Distinct from `/dhcp-client/{ip}` which only sees clients that asked for DHCP.

### M4 · GET /devices

Paginated device list. Query: `?page=1&page_size=50&model=F670L&online=true&pppoe_username=*-customer-*`. Wraps GenieACS NBI `/devices?query=...&projection=_id,...&limit=...&skip=...`. Used by admin UI device discovery.

### M5 · GET /devices/search

Single-device lookup by alternative keys: `?mac=AA:BB:CC:DD:EE:FF` or `?serial=ZTEGC...` or `?pppoe_username=cust-001`. Returns the matching device record (or 404). Useful when IP is unknown — e.g., during onboarding when the customer hasn't been assigned an IP yet.

### M6 · PUT /qos/{ip}

Set bandwidth / QoS profile via TR-069 SetParameterValues on `InternetGatewayDevice.QueueManagement.*` and `Layer3Forwarding.*` (TR-098) or equivalent TR-181 paths. Body: `{download_kbps, upload_kbps, profile_name?}`. Alternative path to RADIUS-attribute-based throttling for CPEs that support TR-069 QoS.

### M7 · GET /wifi-stats/{ip}

Reads `LANDevice.1.WLANConfiguration.*.Stats.*` and `RadioConfiguration.*.{Channel,TxPower,...}`. Returns per-radio: channel, tx power, packet errors, retransmits, beacon interval. Used by WiFi optimization recommendations and customer support "my wifi is slow".

### M8 · PUT /bridge-mode/{ip}

Toggle CPE bridge mode (PPPoE termination on customer router instead of CPE). Body: `{enabled: true|false}`. Sets `WANConnectionDevice.1.WANPPPConnection.1.Enable` and related params. Common deployment scenario where the customer has their own MikroTik / Mesh AP.

## Phase 4 — LOW priority endpoints (10)

These are customer-facing self-service features. Lower priority because most ISPs handle them via a separate customer portal that may or may not use the agent. Group them together for v2.2.0 since they share infrastructure.

| # | Method | Path | TR-069 paths (TR-098) | Body |
|---|---|---|---|---|
| L1 | PUT | `/port-forwarding/{ip}` | `WANConnectionDevice.*.WANIPConnection.*.PortMapping.*` | `{rules: [{name, protocol, external_port, internal_ip, internal_port, enabled}]}` |
| L2 | PUT | `/dmz/{ip}` | `WANConnectionDevice.*.WANIPConnection.*.X_HW_DMZ.*` (vendor-varies) | `{enabled, host_ip}` |
| L3 | PUT | `/ddns/{ip}` | `Services.X_HW_DDNS.*` (vendor-varies) | `{enabled, provider, hostname, username, password}` |
| L4 | PUT | `/wifi-schedule/{ip}` | `LANDevice.1.WLANConfiguration.*.X_HW_WiFiTimer.*` (vendor-varies) | `{schedules: [{day, start_time, end_time, enabled}]}` |
| L5 | PUT | `/mac-filter/{ip}` | `LANDevice.1.WLANConfiguration.*.WLANAccessControl.*` | `{mode: "allow"|"deny", macs: [...]}` |
| L6 | PUT | `/static-dhcp/{ip}` | `LANDevice.1.LANHostConfigManagement.IPInterface.*.DHCPStaticAddress.*` | `{leases: [{mac, ip, hostname?}]}` |
| L7 | PUT | `/ntp/{ip}` | `Time.NTPServer*` | `{ntp_servers: [...], timezone}` |
| L8 | PUT | `/admin-password/{ip}` | `UserInterface.WebPassword` | `{password}` |
| L9 | PUT | `/tags/{ip}` | (GenieACS-side, not TR-069) | `{add: [...], remove: [...]}` — uses NBI `PUT /devices/{id}/tags/{tag}` |
| L10 | GET/PUT/DELETE | `/presets/{name}` | (GenieACS-side, not TR-069) | manage GenieACS provisioning presets via NBI `/presets` |

L2-L5 are vendor-specific. v2.2.0 ships with **TP-Link Huawei ZTE** path support detected by model string; other vendors return `VENDOR_NOT_SUPPORTED` with the same pattern as `OPTICAL_NOT_SUPPORTED`. L7-L8 are TR-098 standard. L9-L10 are GenieACS NBI passthrough.

## Open questions / decisions deferred

1. **Vendor detection for PPPoE/WAN paths** — v2.2.0 hardcodes TR-098 standard paths. v2.3.0 should add the optical-style 5-vendor auto-detect for PPPoE and WAN connection paths (ZTE, Huawei, FiberHome, Realtek, generic TR-181). Track as v2.3.0 issue.
2. **Persistent task tracker** — H5 returns the GenieACS task ID and lets the agent poll. If field experience shows the agent wants a relay-side task store with retention beyond NBI's own, add it in v2.3.0+.
3. **Bulk operations** — apply the same config to N devices in one call. Out of scope for v2.2.0; better implemented at the agent workflow layer with a fan-out activity. v2.4.0+ if there's pull demand.
4. **Auth & RBAC for sensitive ops** — H6 (factory-reset), L8 (admin-password) are particularly sensitive. v2.2.0 relies on the existing global API key middleware. v2.3.0+ may add per-endpoint scopes (`firmware:write`, `factory-reset:execute`) if multi-tenant use cases emerge.
